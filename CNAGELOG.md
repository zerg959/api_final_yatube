Когда вы запустите проект, по адресу http://127.0.0.1:8000/redoc/ будет доступна документация 
для API Yatube. 
В документации описано, как должен работать ваш API. 
Документация представлена в формате Redoc.

Есть распространённый способ проектирования, согласно которому сначала составляют 
документацию, а потом, основываясь на ней как на техническом задании, 
пишут программную часть API. 
Это задание вам предстоит выполнить именно по такой схеме.

Задание 
Ваша задача — дописать код и привести его в соответствие с документацией: 
+++ 1) добавить недостающие модели в приложении posts, 
+++ 2) создать адреса и представления для обработки запросов в приложении api. 

Документация — это ваше техническое задание.
Например, 
+++ 3) в проекте должна быть описана модель Follow, 
+++ 4) в ней должно быть два поля — user (кто подписан) и 
+++ 5) following (на кого подписан). 
Для этой модели в документации уже описан эндпоинт /follow/ и два метода:
-) GET — возвращает все подписки пользователя, сделавшего запрос. 
Возможен поиск по подпискам по параметру search
-) POST — подписать пользователя, сделавшего запрос на пользователя, 
переданного в теле запроса. 
+++ 6) При попытке подписаться на самого себя, пользователь должен получить 
информативное сообщение об ошибке. 
7) Проверка должна осуществляться на уровне API. 
8) Анонимный пользователь на запросы к этому эндпоинту должен получать ответ с 
кодом 401 Unauthorized. 

9) Сейчас ни самой модели Follow, ни обработчиков запросов в коде нет. 
Надо их написать.

Описание проекта: README.md 
10) Выполненный проект должен сопровождаться описанием. 
11) Обязательный минимум — файл README.md, его сохраняют 
в корневой директории репозитория. Обычно он содержит несколько разделов:
12) Описание. Что это за проект, какую задачу он решает, в чём его польза; 
13) Установка. Как развернуть проект на локальной машине; 
14) Примеры. Некоторые примеры запросов к API. 
15) Заполнять README.md удобно прямо на GitHub. В файлах *.md для форматирования текста
(для добавления заголовков, оформления ссылок, выделения текста жирным или 
курсивом) разрешена разметка Markdown. 
Не пренебрегайте ей, это улучшит читабельность вашего текста. 

16) Опишите ваш проект; пример можно подсмотреть в проекте Kittygram: 
возьмите его за основу вашего описания. 

Ключевые моменты 
18) Применяйте вьюсеты. 
19) Для аутентификации используйте JWT-токены. 
20) У неаутентифицированных пользователей доступ к API должен быть только на 
чтение. 
21) Исключение — эндпоинт /follow/: доступ к нему должен предоставляться только
аутентифицированным пользователям. 
22) Аутентифицированным пользователям разрешено изменение и удаление своего 
контента; 
23) в остальных случаях доступ предоставляется только для чтения. 
24) Добавление новых пользователей через API не требуется.

-) Подсказки 
-) Для проверки прав в DRF удобно использовать пермишены. 
-) При необходимости пишите и применяйте собственные классы разрешений. 
-) При описании вьюсетов для некоторых моделей имеет смысл наследоваться от 
собственного базового вьюсета. Не пренебрегайте этой возможностью.
-) Работу с JWT-токенами удобно организовать при помощи библиотеки Djoser, 
но выбор решения за вами.
